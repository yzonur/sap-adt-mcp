# 7 bugs in the ABAP MCP ‚Äî and what they teach you about agent-tool design

While putting Claude through a real ABAP refactor against a ~147 KB class
(`/FGLR/CL_FLEET_TRANSFER` on a real on-prem E4D), seven bugs in
[claude-for-abap](https://github.com/yzonur/claude-for-abap) surfaced inside
one session. None of them showed up in the unit-test suite because they all
sit at the seam between MCP and ADT ‚Äî the layer where the agent's tool calls
meet SAP's REST API. All seven are now fixed (v0.5.4). Here is the write-up,
because the underlying lessons generalise to any tool you build for an agent.

---

## Bug 1 ‚Äî `adt_set_source` body cap blocks large classes

**Symptom.** Writing a 147 KB class via `adt_set_source` is impossible.
The full text doesn't fit in a single MCP tool call.

**Fix.** New tool `adt_set_source_chunked`. Caller acquires a lock with
`adt_lock`, sends the source in chunks under a stable `bufferId` with
sequential `chunkIndex`, and commits with `commit=true`. Server-side buffer
with 10-minute TTL, 4 MB total cap, strict ordering. Same partial-source guard
(Bug 3) applies at commit time. The lock must remain held across all chunks ‚Äî
the chunked write does not silently re-lock between calls.

**Lesson.** Agents don't think in transfer sizes. If your tool's natural
payload can exceed one call, design a multi-call protocol for it from day one
‚Äî don't make the agent discover the cap by failing.

---

## Bug 2 ‚Äî `adt_activate` missing `processRedoneOOSourceVersionOnly`

**Symptom.** In multi-developer scenarios where the object's components are
locked in a separate task under the same transport, activation returns 403
("Object components locked in request and separate task"). The tool offered
no way out.

**Fix.** Added optional `processRedoneOOSourceVersionOnly` to the
`adt_activate` schema. When true it forwards as `isProcessRedoneOOSourceVerOnly=true`
on the ADT activation URL ‚Äî the same parameter ADT in Eclipse sends to recover
from this state. Also exposed `preauditRequested` as an override.

**Lesson.** "Backend supports this query param" and "tool exposes this query
param" are not the same fact. Every parameter ADT in Eclipse uses is one the
agent will eventually need.

---

## Bug 3 ‚Äî `adt_set_source` had no partial-write guard

**Symptom.** PUT on ADT is atomic ‚Äî it replaces the entire include. If the
caller sends a chunk thinking it is a diff, the rest of the include is
silently deleted. A near-miss in an earlier session almost did this to a
3 500-line class.

**Fix.** `detectPartialSource()` rejects input whose first non-comment line
does not start with a recognised top-level ABAP keyword (`CLASS`, `INTERFACE`,
`REPORT`, `PROGRAM`, `FUNCTION`, `FORM`, `MODULE`, `METHOD`, `DEFINE`, `DATA`,
‚Ä¶). Bypass with `acknowledgePartial: true`. Tool description rewritten:
"FULL text ‚Äî atomic replace ‚Äî passing a partial chunk deletes the rest."

**Lesson.** This is the canonical agent-tool failure mode. Agents will
hallucinate that a destructive call is non-destructive, especially when the
parameter is named like a payload. The fix is *not* clearer docs ‚Äî agents skim
docs. The fix is making the destructive call refuse to run on data that does
not look like what the docs promised.

---

## Bug 4 ‚Äî `adt_get_source` had no pagination

**Symptom.** Reading a 154 KB class source can't fit in a single MCP tool
result. The agent falls back to side-channel reads, or never sees the source
at all.

**Fix.** Added `firstLine` / `lastLine` (clamped to bounds) and `onlyMethod`
parameters. The response now includes `totalLines`, `totalBytes`, `firstLine`,
`lastLine`, `scope`, and `truncated`, so the agent can paginate without
guessing. `onlyMethod` slices a `METHOD ‚Ä¶ ENDMETHOD` block by name ‚Äî
case-insensitive, skips declarations in `CLASS ‚Ä¶ DEFINITION` (those have no
`ENDMETHOD`).

**Lesson.** "Return everything" is not a useful default for any read tool that
can occasionally hit a big object. Give the agent a way to ask narrower
questions before it has to.

---

## Bug 5 ‚Äî `adt_lock` stripped the diagnostics from CTS errors

**Symptom.** When LOCK returned 403 on a transport conflict, the agent saw
"lock failed" ‚Äî but the XML body contained `LONGTEXT` (human-readable
diagnosis), `T100KEY-V1..V4` (the *blocking transport's ID*, who owns it, the
suggested resolution). All of that was dropped on the floor by the error
parser.

**Fix.** Extended `parseAdtError` to extract `<entry key="‚Ä¶">` and
`<property name="‚Ä¶">` shapes; pulled out `LONGTEXT` (HTML-stripped) and
`T100KEY-ID/NO/V1..V4`. Surfaced as `error.properties.longText` and
`error.properties.t100.{id, number, vars}`. The agent now sees *exactly which TR
is blocking* and can suggest the right action.

**Lesson.** When you wrap a backend error, the wrapping is where most agent
intelligence is destroyed. A generic "the call failed" is worse than no error
handling at all ‚Äî at least the latter passes the raw body through.

---

## Bug 6 ‚Äî `adt_lock` did not accept `corrNr`

**Symptom.** ADT's LOCK accepts a `corrNr=<TR>` query parameter to scope the
lock to a specific transport. The MCP tool did not expose it.

**Fix.** Added optional `transport` parameter to `adt_lock`. The internal
`acquireLock(client, path, options)` helper now takes `{ accessMode, corrNr }`
(string form still works for backwards compatibility). `adt_set_source` and
`adt_delete_object` also forward `transport` to the LOCK call now, not just to
the PUT/DELETE, for symmetry.

**Lesson.** Same as Bug 2.

---

## Bug 7 ‚Äî Method-level class URI does not exist

**Symptom.** Seven URL variants of `/sap/bc/adt/oo/classes/<class>/includes/method/<method>`
all returned 404.

**Conclusion.** SAP's ADT REST does not expose a method-scope source endpoint.
ADT Eclipse itself fetches the full implementations include and navigates
client-side. The `onlyMethod` parameter added by the Bug 4 fix is the right
answer: server fetches the include, slices the `METHOD ‚Ä¶ ENDMETHOD` block, and
returns just that with line metadata.

**Lesson.** When a backend doesn't expose a granular endpoint, the agent tool
should still expose the granular *operation* ‚Äî slicing client-side is fine, as
long as the agent doesn't have to know that's what's happening.

---

## What gets shipped in v0.5.4

- `adt_set_source_chunked` (new tool)
- `adt_set_source.acknowledgePartial` (new param + guard)
- `adt_get_source.firstLine / lastLine / onlyMethod` (new params + metadata)
- `adt_activate.processRedoneOOSourceVersionOnly / preauditRequested` (new params)
- `adt_lock.transport` (new param) ‚Äî also forwarded by `adt_set_source` /
  `adt_delete_object` internally
- `parseAdtError` extracts `properties.longText` + `properties.t100` from ADT
  exception envelopes

54 new unit tests, all green (141/141 total). No real-system regression
testing yet ‚Äî coming in a follow-up against the E4D system.

## Themes

These bugs cluster into three patterns that every agent-tool author should
own as a checklist:

1. **Destructive calls must refuse plausible-looking-but-wrong input.** A
   guard that costs one regex match is worth more than a paragraph in the
   description.
2. **Don't strip backend diagnostics.** The information density in a SAP /
   any-enterprise-backend error envelope is much higher than in a generic
   HTTP status. Forward it.
3. **Design for payload size up front.** Both read and write paths need
   pagination or chunking before they ship ‚Äî agents will hit the wall at the
   worst moment otherwise.
